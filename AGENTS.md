# Repository Guidelines

## Project Structure & Module Organization
This repository is a Rust workspace defined in `Cargo.toml`. Crates live under `crates/`:

- `crates/leshy-cli/`: command-line entry point in `src/main.rs`
- `crates/leshy-core/`: shared library code in `src/lib.rs`
- `docs/`: project notes such as `docs/project.md`
- `target/`: build artifacts generated by Cargo

Keep new reusable logic in `leshy-core` and keep `leshy-cli` focused on argument parsing, I/O, and orchestration.

## Build, Test, and Development Commands
Use Cargo from the repository root:

- `cargo build`: build the full workspace
- `cargo test`: run unit tests and doctests for all crates
- `cargo run -p leshy-cli -- index --path .`: run the CLI against the current repository
- `cargo fmt`: format all Rust sources
- `cargo clippy --all-targets --all-features -D warnings`: catch lint issues before review

Run `cargo fmt` before opening a PR. At the moment, `cargo fmt --check` is sensitive to empty files, so make sure new source files contain valid, formatted Rust code.

## Coding Style & Naming Conventions
Follow standard Rust style: 4-space indentation, `snake_case` for functions/modules, `PascalCase` for types and enums, and `SCREAMING_SNAKE_CASE` for constants. Prefer small modules with explicit responsibilities. Use `clap` derive attributes consistently in CLI code and keep validation helpers close to the argument types they support.

Add comments only when:
1. A function is considered a public API
2. A piece of code is complex and comments add necessary information to understand what is happening.

## Testing Guidelines
The workspace currently has no substantive tests; add them as new behavior is introduced. Place unit tests in `#[cfg(test)]` modules beside the implementation, and add integration tests under each crate’s `tests/` directory when behavior crosses module boundaries. Name tests for the behavior they verify, for example `rejects_non_directory_path`.

## Commit & Pull Request Guidelines
The visible history is minimal (`Initial commit`), so use short, imperative commit messages such as `Add repository path validation`. Keep commits focused. PRs should include a brief summary, the commands you ran (`cargo test`, `cargo fmt`, `cargo clippy`), and any sample CLI output when behavior changes.
